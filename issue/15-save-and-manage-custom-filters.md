# Issue #15：自訂 filter 命名／儲存／管理（localStorage）

## 背景／為什麼

#14 做出「條件建構器」後，使用者只能現場組條件、重整就消失。但使用者真正想要的是：把一組條件**取個名字存起來**（例如把 `(anthropic && modelId 含 haiku) or (openai && modelId 含 mini)` 存成「**lite models**」），**未來一鍵套用**直接比較這群輕量入門級模型。

這張在 #14 之上加「命名 ＋ 儲存 ＋ 管理」，資料存在 **localStorage**（瀏覽器本機，免後端、免帳號，符合本專案現有「不做後端寫入 / 不做帳號」的範圍）。

## 範圍

**要做：**
- 在 #14 條件建構器旁，新增「**命名並儲存目前條件**」：輸入名稱（如 `lite models`）把當前條件組合存起來。
- 儲存到 **localStorage**：
  - 用清楚、具命名空間的 key（例如 `llm-dashboard.savedFilters`）。
  - 結構帶**版本／schema 標記**（之後條件結構若改版，能判斷並安全處理舊資料）。
  - 存的就是 #14 定義的可序列化條件結構（含名稱）。
- 「**已存 filter 列表**」：列出所有已存 filter，可
  - **選一個套用**（把條件灌回 #14 建構器並過濾表格）；
  - **刪除**；
  - **覆蓋更新**（用同名儲存、或對既有項目更新成目前條件）。
- **持久化**：重整／重開瀏覽器後，已存 filter 仍在並可套用。
- **空狀態**：尚無任何已存 filter 時顯示可讀提示（例如「尚未儲存任何 filter」）。
- **壞資料防呆**：localStorage 內容被手動清掉或格式損壞／版本不符時，前端**不報錯、不破版**，安全退回「無已存 filter」狀態。
- 手機寬度下可操作、不破版。

**不做：**
- 不接後端、**不做跨裝置同步、不做分享連結**（localStorage only）。
- 不做使用者帳號。
- 不改 **#14** 的條件**邏輯**本身（本張只負責命名／儲存／套用／管理那層）。
- 不做檔案**匯出／匯入**（未來要再另開 Issue）。
- 不動長條圖（#13）、modelId／updatedAt 顯示（#12）。

## 驗收條件

- [ ] 能把目前條件組合**輸入名稱後儲存**，該 filter 出現在「已存 filter 列表」。
- [ ] **重整／重開瀏覽器後**，已存 filter 仍在。
- [ ] 點選某個已存 filter 能**套用**並正確過濾表格（條件灌回建構器、結果與該條件一致）。
- [ ] 能**刪除**已存 filter；能用**同名或編輯後覆蓋更新**既有 filter。
- [ ] localStorage 資料被手動清掉／格式損壞／版本不符時，前端不報錯，回到「無已存 filter」狀態。
- [ ] 尚無已存 filter 時顯示可讀的空狀態。
- [ ] 桌面與手機寬度（390px）皆可操作、不破版。

## 預估大小

**中**（偏小；localStorage 讀寫 ＋ 版本/防呆 ＋ 已存清單 UI ＋ 套用回建構器；預估 ~150–220 行有意義變動）。
